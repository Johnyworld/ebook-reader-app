## 언어

- 답변은 항상 한국어로 작성한다.

## 커밋 / PR 규칙

- 커밋 메시지에 Co-Authored-By를 넣지 않는다.
- 기능이 완전히 동작하는 것을 확인하기 전까지 커밋하지 않는다.
- 커밋하기 전에 반드시 코드 리뷰 에이전트가 리뷰한다.
- 커밋하기 전에 인스트루먼트 테스트를 실행할 것인지 물어본다.
- main 브랜치에서 PR 요청 시, 새 브랜치를 생성한 후 main을 origin/HEAD로 되돌린다.

## 주석

- 코드 작성시 의미를 알기 쉽도록 주석을 적극 활용한다.
- 주석은 한국어로 작성한다.

## EPUB 렌더링

- EPUB 페이지네이션은 epub.js를 사용한다. 직접 구현(scrollBy, CSS columns, pageBreaks 등)은 텍스트 잘림 문제로 모두 실패한 이력이 있다.

## 테스트 방침

- 순수 로직(파서, 헬퍼 함수, 마이그레이션 등)은 가능하면 TDD로 진행한다. 테스트를 먼저 작성하고, 테스트를 통과하도록 구현한다.
- UI 코드(Compose, WebView 연동 등)는 구현 후 필요시 테스트를 추가한다.

## e-ink 기기 최적화 룰

- 애니메이션은 구현하지 않는다.
- 회색 배경색은 사용하지 않는다. 배경색은 무조건 흰색만 사용한다.
- 글자 색상을 사용하지 않는다. 대부분 검정색이고, 필요 시 회색을 사용할 수 있다.

## 코딩 지침

LLM이 코딩할 때 자주 저지르는 실수를 줄이기 위한 행동 지침. 필요에 따라 프로젝트별 지침과 병합해서 사용.
트레이드오프: 이 지침은 속도보다 신중함에 무게를 둠. 사소한 작업에는 판단에 맡길 것.

### 1. 코딩하기 전에 먼저 생각하라

**가정하지 말 것. 혼란을 숨기지 말 것. 트레이드오프를 드러낼 것.**

구현하기 전에:

- 가정을 명시적으로 밝힐 것. 불확실하면 질문할 것.
- 해석이 여러 가지일 수 있다면 모두 제시할 것 — 혼자 정하지 말 것.
- 더 단순한 방법이 있다면 말할 것. 필요할 때는 반박할 것.
- 불분명한 게 있으면 멈출 것. 무엇이 혼란스러운지 짚고, 물어볼 것.

### 2. 단순함이 먼저

**문제를 푸는 데 필요한 최소한의 코드만. 추측성 코드는 금지.**

- 요청받지 않은 기능은 만들지 말 것.
- 한 번만 쓰일 코드를 추상화하지 말 것.
- 요청되지 않은 "유연성"이나 "설정 가능성"을 넣지 말 것.
- 일어날 수 없는 시나리오에 대한 에러 처리는 하지 말 것.
- 200줄 짠 코드가 50줄로 가능하다면, 다시 쓸 것.

스스로에게 물어볼 것: "시니어 엔지니어가 이걸 보고 과하게 복잡하다고 할까?" 그렇다면 단순화할 것.

### 3. 수술하듯 정밀하게 수정하라

**꼭 건드려야 할 것만 건드릴 것. 본인이 만든 흔적만 정리할 것.**

기존 코드를 편집할 때:

- 인접한 코드, 주석, 포매팅을 "개선"하지 말 것.
- 망가지지 않은 것을 리팩토링하지 말 것.
- 본인이라면 다르게 할지라도, 기존 스타일을 따를 것.
- 관련 없는 죽은 코드(dead code)를 발견하면 언급만 할 것 — 삭제하지 말 것.

변경으로 인해 고아(orphan) 코드가 생긴 경우:

- 본인의 변경 때문에 사용되지 않게 된 import/변수/함수는 제거할 것.
- 요청받지 않은 한, 원래 있던 죽은 코드는 제거하지 말 것.

기준: 변경된 모든 줄은 사용자의 요청과 직접 연결되어야 한다.

### 4. 목표 기반 실행

**성공 기준을 정의할 것. 검증될 때까지 반복할 것.**

작업을 검증 가능한 목표로 바꿀 것:

- "유효성 검사 추가" → "잘못된 입력에 대한 테스트를 작성하고, 통과시켜라"
- "버그 수정" → "버그를 재현하는 테스트를 작성하고, 통과시켜라"
- "X 리팩토링" → "리팩토링 전후로 테스트가 통과하는지 확인하라"

여러 단계로 이뤄진 작업이라면, 간단한 계획을 명시할 것:

```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

성공 기준이 명확하면 혼자서도 반복 작업이 가능하다. 기준이 약하면("그냥 동작하게 해줘") 계속 추가 설명을 요구해야 한다.

이 지침이 잘 작동하고 있다는 신호: diff에 불필요한 변경이 줄어든다, 과도한 복잡성으로 인한 재작성이 줄어든다, 실수를 한 뒤가 아니라 구현 전에 명확화 질문이 나온다.
